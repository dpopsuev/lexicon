# User Stories

**Core idea:** Express requirements from the user's perspective, not the developer's. Focus on value, not implementation.

## Format

```
As a [role],
I want [goal/desire],
so that [benefit/reason].
```

## INVEST Criteria

Good stories are:
- **I**ndependent — can be delivered standalone
- **N**egotiable — not a contract, a conversation starter
- **V**aluable — delivers value to someone
- **E**stimable — small enough to estimate
- **S**mall — fits in one iteration
- **T**estable — has clear acceptance criteria

## Djinn Alignment

- Parchment Need = the problem (user's perspective)
- Parchment Spec = the solution design
- Gherkin scenarios = testable acceptance criteria
