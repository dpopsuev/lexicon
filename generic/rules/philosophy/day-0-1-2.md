# Capability Tiers — Day 0 / Day 1 / Day 2

**Day 0 boots. Day 1 works standalone. Day 2 enriches. Every feature must work at Day 1.**

Originally from SRE and Operator frameworks (Red Hat Day 1/2 Operations). Applied here to any system that integrates with external services.

## The Three Tiers

**Day 0 — boots with zero configuration.** The binary starts. No config file, no external services, no network required. If it cannot start without something, that something must be bundled or eliminated.

**Day 1 — fully functional standalone.** All core features work without external services. Built-in defaults cover the common case. The system delivers its primary value without any integrations connected.

**Day 2 — enriched by external integrations.** External services (databases, APIs, MCPs, plugins) connect and upgrade Day 1 capabilities. Disconnecting an integration degrades quality but does not break functionality.

<example>
Day 0: the CLI binary starts and prints help with `--help`.
Day 1: the linter runs locally on any file without a language server connected.
Day 2: a language server connects and adds hover docs and go-to-definition.
Day 2 down → Day 1 still works. Day 1 down → nothing works.
</example>

## Rules

1. **Every feature must work at Day 1.** If a feature requires an external service to function at all, redesign it to have a built-in fallback.
2. **Day 2 enriches, never gates.** External service unavailable → the feature degrades gracefully, not fails hard.
3. **Day 0 is the smoke test.** If the binary fails to start with zero config, something is wrong.
4. **Same interface, different implementation.** Built-in and external implementations of the same capability implement the same interface. The caller does not know which is active.

## Anti-Patterns

- **Hard dependency on external service.** Feature only exists when a specific service is connected — no fallback.
- **Conflating Day 1 and Day 2.** Shipping a feature with no built-in default because "the MCP server handles it."
- **Day 0 requires environment variables.** Binary exits with an error if a config var is absent. Provide defaults.

## Complements

- `ports-and-adapters` — the port interface enables swapping Day 1 built-in for Day 2 external implementation
- `integrate-early` — wire Day 1 first; enrich with Day 2 once the interface is stable
- `skills-and-mcps` — MCPs are Day 2 enrichment; skills must work without them
